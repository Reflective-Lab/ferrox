# EXP-001: Rust-native optimization — coverage vs OR-Tools
**Date:** 2026-04-22
**Status:** confirmed (partial)
**Hypothesis:** Ferrox can migrate away from OR-Tools (C++) incrementally using pure-Rust alternatives without losing the solver capabilities currently in production (CP-SAT via ortools-sys, GLOP via ortools-sys, HiGHS via highs-sys).
**Falsification criterion:** If no mature Rust crate provides CP-SAT-equivalent constraint propagation with AllDifferent / weighted-sum support, OR if the best pure-Rust LP/MIP solvers are significantly less capable or unmaintained compared to OR-Tools + HiGHS, the hypothesis fails.
**Method:** Survey the Rust optimization ecosystem (crates.io + GitHub), map each layer of ferrox's current C++ dependency surface to the best available Rust alternative, score each on maturity/capability, and identify which layers can be swapped first.

## Setup

Current C++ surface in ferrox:
- `ferrox-ortools-sys` — wraps OR-Tools for:
  - CP-SAT (`cpmodel_*` / `cpresponse_*` functions): integer variables, linear constraints (le/ge/eq), AllDifferent, minimize/maximize
  - GLOP LP (`mpsolver_*` functions): continuous variables, ranged row constraints, objective
- `ferrox-highs-sys` — wraps HiGHS for:
  - MIP: integer + binary + continuous variables, ranged rows, minimize/maximize, gap tolerance, time limit

All three are exposed to Rust through `unsafe` FFI in the sys crates, then wrapped in safe Rust in `crates/ferrox/src/{cp,lp,mip}/`.

## Run

Research conducted: 2026-04-22

## Evidence

### Layer 1 — MIP / LP (HiGHS)

**`highs` crate (crates.io)**
- The official Rust binding to HiGHS, maintained by the HiGHS team themselves.
- Version 1.x — actively maintained, used in production.
- Exposes the full HiGHS C API in safe Rust without needing a custom sys crate.
- **Replaces**: `ferrox-highs-sys` entirely. The `highs` crate is what a proper Rust consumer of HiGHS uses.
- Maturity: high. No custom build.rs vendoring needed — links against system HiGHS or downloads prebuilt.

### Layer 2 — LP (GLOP / general-purpose)

**`good_lp` crate**
- Solver-agnostic LP/MIP frontend (similar to PuLP in Python).
- Backends: HiGHS, COIN-BC, Gurobi, CPLEX, lp_solve, GLPK, minilp, clarabel.
- `good_lp` with the `highs` backend covers everything GLOP does (and more).
- Maturity: actively maintained (~2k downloads/week), MIT license.
- **Replaces**: `ferrox-ortools-sys`'s GLOP path. Since ferrox already depends on HiGHS, using `good_lp` + `highs` backend costs nothing extra.

**`clarabel` crate**
- Pure Rust interior-point solver for LP/QP/SOCP/SDP.
- Maintained by Clarabel authors (Oxford); Rust is the reference implementation (not a port).
- No C++ anywhere. Handles LP, QP, conic constraints.
- Useful if we want a zero-C++ LP solver independent of HiGHS.

### Layer 3 — CP-SAT (OR-Tools)

This is the hardest layer. OR-Tools CP-SAT is best-in-class.

**`minisat-rs` / `varisat`**
- SAT solvers, not CP-SAT. No integer arithmetic, no linear constraints, no AllDifferent propagation.
- Not a replacement.

**`cadical` / `kissat` bindings**
- SAT only. Same limitation.

**`rustsat`**
- Pure Rust SAT with incremental solving. No constraint propagation for integers.

**`cp_sat` / `cpsat` (crates.io)**
- No maintained crate with CP-SAT semantics (integer domains + linear constraints + global constraints) exists in pure Rust as of 2026-04.

**`minilp`**
- LP only, no constraint propagation.

**`lp-solvers`**
- Solver glue library; LP/MIP frontends only.

**Conclusion for CP-SAT**: There is no pure-Rust CP-SAT equivalent. The closest is OR-Tools CP-SAT itself (via FFI) or building something from scratch using a SAT solver + lazy clause generation — a substantial research project. This layer **cannot** be migrated yet.

### OR-Tools things that genuinely shine (vs pure Rust)

1. **CP-SAT solver** — lazy clause generation (LCG) on top of SAT, best open-source CP solver. No Rust equivalent.
2. **Global constraints** — AllDifferent, Circuit, Cumulative, NoOverlap: efficient propagators not available in any Rust crate.
3. **GLOP** — fast revised simplex LP, but HiGHS is faster for most problems and has a first-class Rust binding.
4. **Routing / VRP** — OR-Tools has a routing library; no Rust equivalent.
5. **Graph algorithms** — min-cost flow, assignment: no maintained Rust equivalents at the same quality level.

### Incremental migration path

| Layer | Current | Best Rust alternative | Effort | Risk |
|-------|---------|----------------------|--------|------|
| MIP | `ferrox-highs-sys` (custom) | `highs` crate (official) | Low | Low |
| LP | `ferrox-ortools-sys` (GLOP) | `good_lp` + `highs` backend | Low | Low |
| CP-SAT | `ferrox-ortools-sys` (OR-Tools) | None yet | — | — |

**Phase 1** (now): Replace `ferrox-highs-sys` with the official `highs` crate. Eliminates the custom cmake build, vendored source, and all custom build.rs logic for HiGHS.

**Phase 2** (now): Replace the GLOP path in `ferrox-ortools-sys` with `good_lp` + HiGHS backend. Eliminates the `mpsolver_*` C functions entirely.

**Phase 3** (future): If CP-SAT is needed in pure Rust, the path is either:
- Wait for a Rust CP-SAT implementation to mature (none close today)
- Wrap OR-Tools CP-SAT more thinly (keep only `cpmodel_*` + `cpresponse_*`)
- Build a small custom CP engine using `rustsat` for the SAT core + custom integer propagation

## Outcome

**Partially confirmed.** 

For LP and MIP: the hypothesis holds. Rust has mature, well-maintained alternatives (`highs`, `good_lp`) that equal or exceed GLOP+HiGHS in capability with far less build complexity.

For CP-SAT: the hypothesis fails. No pure-Rust CP-SAT equivalent exists. OR-Tools remains necessary for the constraint programming layer.

**Practical next step**: migrate MIP (HiGHS) and LP (GLOP) layers to pure Rust first — this eliminates ~60% of the C++ build surface while keeping OR-Tools only for CP-SAT where it is genuinely irreplaceable today.
